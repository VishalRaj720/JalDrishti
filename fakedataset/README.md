# Synthetic Dataset (Month 4)

This folder is **populated by `python -m ml.synthetic`** (run from `backend/`).
Files here are git-tracked so the demo pipeline is reproducible without re-fitting the copula.

| File | Producer | Consumer |
|------|----------|----------|
| `synthetic_wells.csv` | `ml.synthetic.generate()` | `ml.load_synthetic` (creates `monitoring_wells` rows) |
| `synthetic_water_samples.csv` | `ml.synthetic.generate()` | `ml.load_synthetic` (creates `water_samples` with `synthetic=True`) |
| `synthetic_metadata.json` | `ml.synthetic.generate()` | provenance record (seed, method, timestamps) |

## Why synthetic data

ISR-specific uranium contamination data is sparse in CGWB/India-WRIS. To keep
the Month 4 baseline classifier from collapsing to the majority "safe" class,
we augment with **500 synthetic samples** generated by a Gaussian copula fit
to the real chemistry distribution, with conditioning that biases ~25% of rows
into the WHO-unsafe regime.

Synthetic rows are flagged in DB via `water_samples.synthetic = TRUE` and
linked to a `data_sources` row of type `synthetic_copula`, so they can be
separated from real rows at evaluation time:

```sql
-- Train on real + synthetic, evaluate on real only
SELECT * FROM water_samples WHERE synthetic = TRUE;   -- training augmentation
SELECT * FROM water_samples WHERE synthetic = FALSE;  -- holdout / real-world eval
```

## Removing synthetic data

```sql
DELETE FROM water_samples WHERE source_id IN (
    SELECT id FROM data_sources WHERE source_type = 'synthetic_copula'
);
DELETE FROM monitoring_wells WHERE source_id IN (
    SELECT id FROM data_sources WHERE source_type = 'synthetic_copula'
);
```
